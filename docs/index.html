Cognee, doing real work

Four calls. None of them decorative.

Every fact carries a confidence score — reinforced on success, penalized on failure, visualized directly as node color in the live memory graph: cyan is trusted, amber is decaying, red is stale.

remember()
Once per session · session-end, batched
Writes all of that session's observations to Cognee in a single add() + cognify() pass — not after every action. That's a fix, not the original design (see Decisions below).
recall()
Before every planning step · cached per room
The real text Cognee returns is shown live in the UI, unsummarized. In fallback mode (no API key), this node is skipped entirely — the deterministic planner doesn't need it.
improve()
After every action · based on real outcome
Confidence rises +0.2 (capped at 1.0) on success, falls −0.5 (floored at 0.0) on failure. Unit-tested directly in tests/test_memory_ops.py — 5 of the 7 tests there are this scoring logic alone.
forget()
On the drift-correction path only
Local confidence removal is guaranteed. The underlying Cognee graph-node prune is attempted, and its real success/failure is returned as an auditable dict — never silently assumed.
Architecture

A real, compiled LangGraph StateGraph

Six nodes and one conditional loop-back edge — verified end-to-end with the deterministic fallback planner, no API keys required, before this was written down as a claim.

Engineering decisions

Fixed after breaking, not designed in from day one

Each of these came from watching the system fail during development — not from anticipating the failure in advance.

Two Groq models, not one

llama-3.3-70b-versatile plans (few, short calls). llama-3.1-8b-instant handles Cognee's internal extraction (frequent calls, no frontier reasoning needed) — on a separate 500K/day pool.

found via: rate-limit exhaustion in dev

remember() batched to session-end

The first version called it after every action. cognify() re-resolves the whole graph and runs LLM extraction on every call — that burned the entire daily Groq budget in 2–3 sessions.

found via: budget burned in 2–3 runs

Ordered, single-consumer event queue

An earlier version fired WebSocket events with plain asyncio.create_task() calls. Rapid events could arrive at the browser out of order — a real race condition, not a hypothetical one.

found via: out-of-order UI events
Try it yourself

No hosted link — and that's deliberate

Cognee's memory lives in local SQLite + LanceDB + Ladybug files. Free hosting tiers have ephemeral filesystems that wipe on redeploy or sleep — which would silently break the entire "memory persists across sessions" premise this project exists to demonstrate. So it runs locally, and a demo video stands in for a hosted link.

Stated plainly, not buried

What this system doesn't do

The honest version of a limitations section — not because a rubric asks for it, but because pretending otherwise would make the memory claim less believable, not more.

—

Symbolic simulation, not a physical robot. The perceive → recall → plan → act → correct → finalize graph is designed to port to a real controller; a short solo build window intentionally excluded physics and hardware risk.

—

The LLM planner is the least deterministic part. Forced tool-calling schemas, pre-execution state validation, capped retries, and a fully unit-tested deterministic fallback planner all exist specifically to contain this.

—

tidy_kitchen is contextual by design. It only meaningfully tests cleanup as a follow-up to a make_coffee session in the same house instance — on a fresh house, the task starts trivially satisfied.

—

forget()'s Cognee-side prune is best-effort. Local confidence removal always succeeds; the underlying graph-node deletion depends on the installed Cognee version's API surface, and its real outcome is returned, not assumed.

Tech stack

Every layer, and why it's there

Simulation
Pure Python dataclasses
Deterministic house/room/object model, partial observability, zero external deps.
Orchestration
LangGraph StateGraph
6 real nodes, 1 conditional loop-back edge, compiled once and reused.
Planning
Groq llama-3.3-70b-versatile
Forced tool-calling schema — the planner can't emit an invalid action shape.
Fallback planning
Deterministic explorer
Hand-rolled, no LLM, no network — unit-tested against a real house.
Memory
Cognee 1.2.2
SQLite + LanceDB + Ladybug — zero-setup local stack, remember/recall/improve/forget.
Memory extraction
Groq llama-3.1-8b-instant
Routed via LiteLLM's groq/ prefix — separate rate-limit pool from the planner.
Embeddings
Fastembed
BAAI/bge-small-en-v1.5, 384-dim, local, no API key required.
Backend
FastAPI + WebSocket
Ordered single-consumer event queue, /ask and /memory/graph endpoints.
Frontend
Vanilla JS + D3.js
Live map, force-directed memory graph, session comparison chart.
